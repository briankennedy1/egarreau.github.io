I think the biggest thing I learned this week was that a lot of the stuff I thought was super fancy and complicated is actually pretty straightforward, it just has a lot of complicated rituals and incantations around it so it seems scary. Version control just means a way of tracking changes. Git lets you track changes to your project systematically, and GitHub lets a whole lot of scattered people track all their changes at once. This is pretty clearly a good thing, in order to not muddy up the code and make sure that the live version is the version that works, and also allow anyone to go back to a previous version if something new happens to break everything.

The main reason git (and GitHub) seem super complicated and strange is that in order to use it (at least, in order to really understand it), you have to pray to the great goddess Unix, using one of Her shells -- high priests that translate between Her divine thoughts and us mere mortals. There is much argument about which shell is the best, but here at The DBC School of Witchcraft and Wizardry we are followers of bash. I am a brand new devotee of bash so I am not yet super familiar with his ways, but so far my initiation has gone smoothly. Git is just one way in which Unix performs Her work, but it is an important and powerful way. To pray to Her you recite incantations to bash, and if spoken correctly (and if the moon cycle is right) She will grant your wish. You must first tell bash what sort of miracle you are praying for (a git miracle, for example), and then the specific action you want that miracle to manifest, and the target of your miracle. I'll give you an example, but first I'll tell you a little more about the relationship between git and GitHub.

When you use git locally on your computer, there are three stages to every change you make: working, staged, and committed. Working is when you're making changes in your text editor, staged is when you notify the high priest bash that you have changes ready to be saved, and committed is when you pray to bash to save your changes. Through the miracle of git, this not only saves your changes but also keeps track of every change you've made previously, in case you screw the pooch and need to roll back your work to a previous version. Yea, the miracle of git does truly enable time travel. Praise Unix.

[I realize this extended metaphor is imperfect, since you don't actually have to use Unix to use git, and the two aren't that closely related. I'm learning; humor me.]

So as an example, for you new initiates out there, let's say I have this blog post saved on my computer as an html file called blog1.html (because I am very creative). When I am ready to record the changes I've made, I pray to bash:

git add blog1.html
git commit -m "update blog post"

In the first line, I pray to bash for a git miracle, and indicate specifically that I want the miracle to ready my changes to blog1.html for saving. In the second line, I indicate my hope for a second git miracle, this time to "commit" the changes (save them) with the message "update blog post" as and indicator for what this change does.

These constant prayers may seem greedy or demanding, but Unix is a truly giving digital deity. Praise Unix.
