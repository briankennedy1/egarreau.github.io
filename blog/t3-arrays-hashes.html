A Few of My Favorite Things

Information Organizational Structures and Leslie Knope

April 2nd, 2015

Ruby has two important organizational structures: arrays and hashes. They are both actually just fancy lists.

An array is a numbered list. You can iterate through the list to look for a particular item, or if you know the number of an item you can go straight to it. For example, here's a ranking of colors in order of my favorites:

  1. purple
  2. blue
  3. yellow
  4. green
  5. red
  6. orange

Accessing this list in Ruby is sort of like calling a phone number with an automated messaging system. The kind robot voice says, "Welcome to Evangeline's Favorite Colors. Please listen closely, as our options have changed. If you know your party's extension, please dial it now." If want purple and you know me well enough to know that it's my favorite color (by, for example, glancing at my twitter profile or passing me on the street) you can dial 0 without listening to all the other options. If you want to know what my least favorite color is, you can dial 5 to get orange without listening to the options. Otherwise, you can listen to the operator iterate through the list to know which number to dial.

Here's how this would look in Ruby:

favoriteColors = ['purple', 'blue', 'yellow', 'green', 'red', 'orange'] #This creates the list of my favorite colors

bestColor = favoriteColors[0] #This assigns the variable bestColor with the first item in the list

puts bestColor #This prints 'purple'

worstColor = favoriteColors[5] #This assigns the variable worstColor with the last item in the list. If you didn't know how long the list was, you could also use favoriteColors[-1] to access the last item, regardless of length.

puts worstColor #This prints 'orange'

for i in 0..5 do
  if favoriteColors[i] == 'green'
    greenRank = i
  end
end
#Just like listening to the robo-operator, this loop goes through the list in order to check for green. When it finds green, it assigns green's index (or list number) to a variable called greenRank

puts greenRank #This prints 3

So now we know how to make a numbered list and extract information from it. But what if I want to make a collection of my favorites in a bunch of different categories? I don't want to rank them, I just want to list my favorite color, holiday, fruit, vegetable, and fictional female character.

Can I do this with an array? Sort of. It would look like this:

favoriteThings = ['purple', 'Springween', 'orange', 'avacado', 'Leslie Knope']

But this list is pretty meaningless, right? We know from the name that it's a list of my favorite things, but in what categories? Avocado could be my favorite flavor of milkshake. Leslie Knope could be my favorite alternate universe candidate for president in 2016. And god forbid, orange could be mistaken for my favorite color, when we all know it's my LEAST favorite color. There HAS to be a better way!

This is where hashes come in. If an array is like a robo-operator, we can think of a hash as a shelf of labeled boxes. So my "Favorite Things" shelf would have five boxes, labeled color, holiday, fruit, veggie, and fictional heroine. The color box would have purple in it, the holiday box would have Springween, the fruit box would contain orange, the veggie box avocado, and the heroine box the inimitable Leslie Knope. You can make your way across the shelf, browsing the labels and opening the boxes to find what you're looking for.

If you're familiar with programming, you may have heard a similar box metaphor used to explain how variables work. So why can't we just do the same thing with variables and arrays instead of using a hash? We sort of can -- it would look like this:

color = 'purple'
holiday = 'Springween'
fruit = 'orange'
veggie = 'avocado'
heroine = 'Leslie Knope'

favoriteThings = [color, holiday, fruit, veggie, heroine]

That would function approximately the same as a hash, at least at my level of understanding -- though I am fairly confident there are a lot of specialized and helpful things you can do with a hash that just aren't possible with an array of variables. But even putting aside the technical limitations of this setup, it's also harder to read and requires more effort. In addition, the get information out of that list, you would need to know the order of the variables. By contrast, building a hash is simple and easy to understand for anyone who is reading your code:

favoriteThings = {
 :color => 'purple',
 :holiday => 'Springween',
 :fruit => 'orange',
 :veggie => 'avocado'
 :heroine => 'Leslie Knope'
}

Now to find out my favorite holiday, you would simply type favoriteThings[:holiday] and the hash would return 'Springween'. You don't need to iterate through the whole list, and you don't need to know where in the list :holiday is stored.

Neither arrays nor hashes are intrinsically better for structuring information; they can each be used best for different purposes. Like most tools, they have their own advantages and disadvantages depending on the scenario. But I think we can all agree: avocados are delicious.

Note: upon cursory Wikipedia-based research I have discovered that avocados are technically a berry. I stand by my preference.
